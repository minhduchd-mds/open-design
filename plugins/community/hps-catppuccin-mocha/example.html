The problem

Every dashboard green.
Checkout still slow.

A single checkout touches nine services. Each one reports a healthy p99 — but queueing, retries and serialization between them add up to 400ms nobody owns.

  • Per-service metrics can't see inter-service wait time
  • Log correlation by timestamp breaks under retries
  • On-call pages route to the wrong team 4 times out of 10

Architecture

Four stages, one pipeline

Instrument

eBPF agent per node captures spans at the socket layer — no SDK required.

Collect

OTLP-compatible collectors batch and compress; 4MB/s per node ceiling.

Correlate

Trace IDs propagate via W3C traceparent; retries fold into one tree.

Alert

SLO burn alerts link straight to the slowest exemplar trace.

The counter pseudo-element numbers the cards automatically. If anyone asks: OTLP-compatible means existing OpenTelemetry SDK spans merge into the same trace tree.
