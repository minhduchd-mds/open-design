// SHT 01 · general notes

General notes

Every component on this sheet obeys four construction rules. Anything that cannot is redrawn until it can.

  • Workers are stateless — kill any of them at any time
  • All cross-domain traffic rides the replicated log
  • Policy is data, not code — hot-reload, never redeploy
  • Each domain runs degraded-but-alive on its own
ConsensusRaft ×5 — survives two voter losses
PlacementRendezvous hash — no rebalance storms
TransportgRPC + mTLS — certs rotate every 24h
StateLog only — snapshots every 5 min
Two-column layout: rules as a checklist on the left, a dashed spec table on the right — the table reads like a materials schedule on a real drawing.

// SHT 03 · cutover sequence

Four phases, rollback lines drawn first

Shadow

Rev C runs beside rev B, replaying the live log. Two weeks, zero traffic.

Mirror

Reads served by C, writes still land on B. Diffs alarmed at 0.01%.

Cutover

Domain by domain, smallest first. One command back to B at any point.

Retire

B kept warm for 30 days, then decommissioned with its snapshots archived.

Process steps with bracketed [01]–[04] counters, like detail callouts on a drawing. The rollback path is stated inside every phase.

// SHT 03 · revision delta

Rev B vs Rev C

Rev B — as built

Chatty & coupled

  • Per-request appends to the log
  • Policy compiled into the scheduler binary
  • Domain failure pages a human at 3am

Rev C — this drawing

Batched & autonomous

  • Appends batched per domain, 2.5× headroom
  • Policy as hot-reloaded data
  • Domains degrade alone, recover alone
